Technical Evidence – Engines, Safety & Tests

This page summarises the real technical backbone of Elisence: what is already built and tested across 17 phases, where WORM & safety engines are live, and how ministries, clinics and investors can request deeper evidence packs.

We do not claim that all 17 phases are “finished”. Instead, we clearly show: which phases are fully implemented and tested, which ones have core engines already built, and which phases are still in early implementation or roadmap stage.

1. Phase-by-Phase Technical Status (17 Phases)

For each phase, we indicate what is built, what kinds of files and artefacts exist, and how far tests and safety layers have progressed. Detailed file lists and paths are provided in private technical packs on request.

  • Phase 1 – Core Logs & Events Built & Tested

    Core event engine for Elisence – captures structured health events and timelines.

    • Built: Event schemas, logging engine, core routing and helpers.
    • Files & artefacts: Main engine module, event models, helper libraries, test suite.
    • Tests & safety: Pytest suites passing for event correctness, ordering and basic safety checks.
    • Examples: Used as the backbone for later phases (GLP-1, diabetes, Women+, AHE).
  • Phase 2 – Weight & GLP-1 Calibration Built & Tested

    GLP-1 and weight calibration engine – turning raw logs into structured journeys.

    • Built: Weight/GLP-1 models, calibration logic, event normalisation.
    • Files & artefacts: Engine module, schemas, calibration helpers, end-to-end tests.
    • Tests & safety: Multiple pytest scenarios (dose changes, weight trends, edge cases) passing.
    • Examples: Feeds structured signals into AHE (Phase 8) and Diabetes (Phase 9).
  • Phase 3 – Behaviour & Dopamine / Care Platform Built & Tested

    Care / weight-loss platform engine with behaviour & dopamine logic and real UI.

    • Built: Weight Core v1, schedules, intake/check-in/SOS logic, dashboard UI.
    • Files & artefacts: Engine module, schemas, /v3/ui dashboard HTML/JS, tests.
    • Tests & safety: All core tests (including 11+ dedicated weight-core tests) passing.
    • Examples: Real dashboard UI with cards (Schedules, Intake, Check-In, SOS, Medical, GP, Pharmacy, etc.).
  • Phase 4 – WORM / FHIR / Provenance Built & Tested

    Regulatory backbone – WORM audit, FHIR mapping foundations and provenance.

    • Built: WORM Event Audit Log v2, FHIR-ready mapping helpers, provenance export paths.
    • Files & artefacts: Main Phase 4 engine (e.g. main_phase4_elisence.py baseline), WORM JSONL file (e.g. worm_audit_v2.jsonl), export endpoints.
    • Tests & safety: WORM on/off tests, export tests, provenance validation and health checks are passing.
    • Examples: JSONL WORM logs with trace IDs and export manifests – ready for ministry/regulator review.
  • Phase 5 – Elisa Companion Layer Built & Tested

    Core Elisa companion experience – safe, non-diagnostic coach that sits on top of the engines.

    • Files & artefacts: Companion engine modules, i18n packs, safety/tone helpers, tests.

2. Evidence Types – UI, WORM Logs, Tests & APIs

Beyond the engines themselves, Elisence already has concrete, verifiable technical evidence:

  • Real UI: Phase 3 care dashboard, Phase 7 Women+ flows, Phase 8 AHE views, Phase 9 diabetes charts and Phase 10 messaging / profile UI, plus the public Elisence website and Elisa chat widget.
  • WORM logs (JSONL): WORM Event Audit Log v2 JSONL files (for example, Phase 4 & Phase 8 evidence) with trace IDs, export manifests and signatures suitable for regulators and ministries.
  • Pytest results: Dozens of tests across engines – including full green suites for Women+ (Phase 7), AHE risk engines and Calm Now (Phase 8), diabetes charts (Phase 9), and multiple safety and token tests.
  • API & docs: FastAPI-based services with /docs and /openapi.json live for multiple phases, including health checks and phase-specific routes (e.g. AHE, WORM export, diabetes charts, messaging).

Screenshots, screencasts, sample WORM JSONL files, pytest logs and OpenAPI exports are included in private technical packs when needed for ministries, clinics or serious investors.

3. Accessing Detailed Technical Evidence Packs

For transparency and safety, we separate public information from deeper technical details:

  • Public downloads: High-level PDFs (for example, the Qatar summary PDF) are available on the Downloads page.
  • Technical Evidence Packs (private): For ministries, clinics, regulators and investors, we provide phase-specific packs that can include:
    • File and module overviews for the relevant phases.
    • Sample WORM JSONL logs and export manifests.
    • Pytest output (number of tests, pass/skip status, focus areas).
    • OpenAPI definitions and selected UI screenshots/screencasts.
  • Scope honesty: These packs clearly state which phases are fully implemented and tested, which phases have engines but are still evolving (for example, Phase 6 and Phase 13), and which phases are still in roadmap or early build stage (for example, Phases 14–17).
