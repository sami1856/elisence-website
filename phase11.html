Phase 11 — Security & Identity Control

Security Backbone, Passport & Access Control

Phase 11 is the security spine of Elisence. It decides who may do what, from where, and under which passport — and it does so under strict Zero-Trust, replay protection and audit-ready rules designed for national digital health systems.

Core Definition

What Phase 11 Is

Phase 11 is Elisence’s Security & Identity Control Layer. It provides a single, consistent way to verify every request that enters the platform:

  • One canonical X-Passport header for all API calls
  • A single RS256/JWT verifier used by all major phases
  • Strict claim validation for identity, role, scopes and expiry
  • Replay protection so passports cannot be reused
  • Audit logging for both valid and blocked access attempts

Design Principles

Zero-Trust by Design Fail-Closed, Never Fail-Open Single Source of Truth Verifier No Backdoors, No Legacy Headers Government-Grade Security Posture Multi-jurisdiction aware (NHS / GDPR / GCC)

Stakeholder View

Who Depends on Phase 11?

  • Phase 10 social profiles, messages and attachments
  • Voice and accessibility flows (Phase 13 & Phase 14)
  • Kids+, family and carer access paths
  • Ministry dashboards and professional portals

Passport & Header Rules

One Header, One Contract

Phase 11 standardises how identity is presented to Elisence. The platform accepts only a single, well-defined header: X-Passport. All other headers such as Authorization or legacy internal names are rejected.

This simplifies security reviews and makes it clear for ministries and partners: there is exactly one way to authenticate into Elisence, and it is fully documented and testable.

JWT & RS256 Validation

Every passport is a signed JWT using RS256 with a recognised key ID (kid). Phase 11 defines the rules for:

  • Rejecting unsupported algorithms
  • Rejecting missing or unknown kid
  • Rejecting corrupted signatures or broken key sets

If anything is wrong, the system responds with a strict 401 JSON response — never with a silent pass-through.

Mandatory Claims & Zero-Trust Checks

Every Claim Must Make Sense

Phase 11 defines the minimal claims that every passport must carry: version, issuer, audience, subject, role, scopes, timestamps and a unique ID. Types are enforced and missing or malformed values are treated as attempts to bypass rules.

  • Wrong issuer or version → rejected
  • Incorrect audience or scopes → rejected
  • Expired or impossible timestamps → rejected
  • Wrong claim type (e.g. role as list) → rejected

In every case, responses follow a canonical JSON shape for 401/403/503, making behaviour predictable and auditable for external reviewers.

Fail-Closed, Not Fail-Open

If the passport bridge is disabled, misconfigured, or under key-rotation, Phase 11 returns a controlled 503 (bridge not configured) instead of guessing or allowing access. This “fail-closed” behaviour is essential for government-grade deployments.

Replay Protection, Key Rotation & Audit

Stopping Reuse and Forged Sessions

Phase 11 introduces a dedicated replay-protection store based on the jti claim:

  • First use of a passport ID → accepted and logged
  • Second use → blocked with a 403 replay_blocked response

This protects high-value operations such as attachments, social actions or administrative changes from copied links and stolen tokens.

Key Rotation Without Downtime

The design of Phase 11 supports overlapping keys: old and new kid values can be valid for a period, with audit trails for rotation events. This allows ministries and partners to rotate keys without breaking clinical and citizen flows.

WORM Audit Logging

Every critical passport event — valid, invalid or replay-blocked — is written into an append-only (WORM) audit log. Raw JWTs are never stored; instead, only safe metadata is kept:

  • time, reason and outcome of the check
  • which system, route or phase requested access
  • trace IDs that link to higher-level evidence packs

This gives ministries and auditors a clear chain-of-custody for security decisions.

Integration With Phase 10 & Social Content

From Login to Social Actions

Phase 11 works hand-in-hand with Phase 10 (User Profiles & Social Presence). It is responsible for:

  • Issuing passports at registration and login
  • Protecting posts, comments, messages and feed actions via scopes
  • Securing attachment links so they cannot be replayed or reused
  • Returning clear 401/403/503 responses when something is wrong

In short, Phase 10 makes Elisence feel social and alive; Phase 11 makes sure that every social action happens under verified, controlled identity.

Security Backbone for All Phases

Shared Verifier, Shared Rules

Instead of each phase building its own security layer, Phase 11 defines a single verifier that all major engines rely on:

  • Phase 7 (Women+), Phase 8 (AHE), Phase 9 (Diabetes)
  • Phase 10 (Social), Phase 13 (Voice), Phase 15+ (Family, Kids+, Mental Health)

This single-source-of-truth approach reduces risk, simplifies audits and makes it easier to certify Elisence as a whole rather than as a collection of separate apps.

Reference Specification

The PASSPORT_V1_REFERENCE_SPEC document describes the contract in plain language: headers, algorithms, mandatory claims, replay rules and canonical error JSON. It is written specifically to support ministry and regulator reviews.

Algorithm & Analytics Layer (Roadmap)

From Security Events to Safer Systems

On top of the security layer, Phase 11 also defines a future analytics layer for:

  • Telemetry about how passports and scopes are used
  • Behavioural and population-level security metrics
  • Privacy-preserving dashboards built with k-anonymity rules

This algorithmic layer is deliberately treated as a next step: it will only be activated once the core security and verification flows have been fully implemented and verified.

Government & Investor Posture

Safe Foundation for a National Platform

Phase 11 is what allows Elisence to be taken seriously as national infrastructure. It is designed to align with:

  • NHS DTAC and clinical safety-style requirements
  • GDPR / UK DPA for data protection and access control
  • EU MDR boundaries for non-diagnostic digital health tools
  • GCC, Qatar and Saudi PDPL data-residency and privacy frameworks

For investors, it demonstrates that Elisence treats security and governance as first-class citizens, not afterthoughts. For ministries, it shows a clear path to risk-managed deployment at scale.

In one sentence: Phase 11 turns Elisence into a Zero-Trust, government-grade identity system — a security backbone that protects every profile, file, message and clinical engine, and that can be audited, explained and trusted at national scale.
